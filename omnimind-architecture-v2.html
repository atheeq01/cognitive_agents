What Changed & Why

Architecture v2
⟳ Complete Model Shift — org_id → Project Workspaces
The original design used a single org_id to scope all data. The new design replaces this with a Project as the core entity. A single user can belong to many projects with different roles in each. Projects are fully isolated knowledge workspaces — documents, embeddings, chat history, conflict results, and AI memory are all scoped to a project_id, not an org. This enables real-world use cases: a law student working on three research projects, a lawyer managing separate cases, or a PhD student juggling multiple papers — each as an independent workspace.
✗ Original Model (org_id)
  • One org per user — no cross-project membership
  • All documents visible to all org members
  • org_id enforced only in application code
  • No concept of projects, cases, or workspaces
  • No approval workflow for document uploads
  • No role-based permission within an org
  • Single tenant boundary = single blast radius
✓ New Model (project_id)
  • Users belong to many projects simultaneously
  • Different roles per project: Admin / Member / Viewer
  • project_id enforced at PostgreSQL RLS layer — not just app code
  • Each project is an independent AI knowledge workspace
  • Admin-controlled document approval workflow per project
  • Fine-grained permissions: who can upload, query, manage
  • Project-level data purge for GDPR compliance

User & Project Model

New Architecture
👤 User Model
  • One account, many projects. A user has a single identity (user_id, email, name, avatar) across the platform.
  • Project membership is explicit. Being a user does not grant access to any project. Membership must be granted by a Project Admin.
  • Role per project. A user can be an Admin in Project A, a Member in Project B, and a Viewer in Project C — all simultaneously.
  • Project switcher. The UI shows all projects the user belongs to. The active project sets the context for all queries, uploads, and AI operations.
  • No cross-project data. Switching projects resets the entire workspace — search results, document lists, chat history, and knowledge graph all scope to the new project.
📁 Project Model
  • Project = independent knowledge workspace. Each project has its own documents, embeddings, AI memory, conflict history, and chat sessions.
  • Created by one Project Admin. The creator automatically becomes the Project Admin. They can promote others to Admin.
  • Project settings per instance. Each project configures: upload approval required (on/off), allowed file types, max document count, AI analysis schedule, visibility of conflicts to members.
  • Isolated storage paths. All Cloud Storage, Cloud SQL rows, Firestore collections, and Pinecone namespaces are all scoped to project_id at both the application and infrastructure layer.
  • Deletion propagates completely. Deleting a project purges all data across every store — SQL, Firestore, GCS, Redis, embeddings — with a deletion certificate for GDPR compliance.
User Signs In
Firebase Auth issues JWT containing user_id. JWT contains no project_id — that's resolved on project selection, not at login.
Project Selection
Client calls GET /users/me/projects → returns list of all projects with user's role in each. User selects a project. Client stores active_project_id in session.
Every Request
API receives Authorization: Bearer {JWT} + X-Project-ID: {project_id} header. Middleware verifies: (1) JWT valid, (2) user is member of project, (3) user has the required role for this endpoint. Fail any check → 403.
DB Layer
Middleware sets SET LOCAL app.project_id = '{project_id}' on the PostgreSQL session. RLS policies filter every table automatically — even if application code forgets a WHERE clause, the DB rejects cross-project reads.
Storage Layer
All Cloud Storage writes use path gs://omnimind-docs/{project_id}/{user_id}/{file_id}. GCS IAM Conditions enforce this path prefix — files cannot be read or written outside the project's prefix even if the app has a bug.
AI Layer
All Pinecone similarity searches use query(namespace=project_id) — the namespace is a hardware-level boundary that prevents cross-project vector access (defense in depth). Gemini prompts include the project_id and explicitly state the project boundary in the system prompt.

  • All request/response DTOs in apps/api/schemas/ extend pydantic.BaseModel (v2). FastAPI validates all incoming JSON against these schemas automatically.
  • UCD & Claim models in schemas/ucd.py use Pydantic v2 for both in-transit validation and as the canonical data contract between the worker and API layers.

Project Data Isolation

Critical Design

Data isolation is the most critical correctness property of the system. For legal users, a cross-project data leak is a professional ethics violation. The design enforces isolation at four independent layers — failing any single layer does not produce a breach.

Alice can see Smith v. Acme and Climate Policy — but never sees ML Ethics project data. Bob can see Smith v. Acme (Member) and ML Ethics (Admin) — but never sees Climate Policy. Zero cross-project leakage, enforced at 4 independent layers.
Layer 1 — Application RBAC
Every route handler depends on require_project_role(). This checks that the user is a member of the requested project before any query runs. If a user is not a member, the request is rejected with 403 before any DB interaction. This is the first line of defence.
Layer 2 — PostgreSQL Row-Level Security
SET LOCAL app.project_id is called on the DB session before every query. RLS policies use current_setting('app.project_id') to filter rows on every table automatically. Even a raw SQL query by an admin, a developer console, or a future code bug cannot return rows from a different project_id.
Layer 3 — Cloud Storage IAM Conditions
GCS IAM Conditions restrict the service account to read/write only within the path projects/{project_id}/. This is enforced by GCP infrastructure independently of any application code. A bug in the file service cannot read another project's files — GCP rejects the request at the storage API layer.
Layer 4 — Pinecone Namespace Isolation
All vector similarity searches are routed through pinecone.query(namespace=project_id). Pinecone namespaces provide hard API-level isolation — a query in namespace A physically cannot return vectors from namespace B, independent of application code. This is the AI layer's defense-in-depth on top of PostgreSQL RLS.
Isolation Test Requirement: Every deployment must run an automated cross-project isolation test: create two test projects, upload documents to each, attempt to read Project B's documents while authenticated to Project A, and verify 403 at all four layers. This test must pass before any promotion to production.

AI Pipeline — Per-Project Cognitive Engine

Core Innovation
✦ Each Project Has Its Own AI Brain
In the project model, every project maintains a completely independent knowledge base, embedding space, conflict history, and conversation memory. The AI has no knowledge of any other project's documents. When you query inside Project A (Case: Smith v. Acme), the retrieval, reasoning, and conflict detection are scoped entirely to that project's corpus — the PhD thesis project documents are invisible, even if you're a member of both.
⚙ 5-Stage Cross-Modal Contradiction Pipeline (Per Project)
1
Claim Extractor
Per-UCD atomic facts with source spans, modality tag, and speaker_id (audio)
2
Candidate Selector
Pinecone query(namespace=project_id, top_k=5, score>0.80) — all modalities
3
NLI Classifier
ENTAIL / NEUTRAL / CONTRADICTION — mandatory evidence quotes
4
Verifier Pass
Independent LLM — clean context, only the two evidence quotes
5
Severity Score
CRITICAL / HIGH / LOW → human queue if disagreement
1
Claim Extractor — Modality & Speaker Aware
Runs on every approved UCD added to the project. Extracts atomic, falsifiable claims with exact source spans. Audio claims include speaker_id from diarization — critical for legal depositions where two speakers may make opposite assertions legitimately. Each claim is stored in the project's claim table and embedded at the claim level.
Output: {claim, modality, source_span, speaker_id, entity_ids, date_ref} Claim-level embedding stored per project_id No inference — only direct attestations from the document
2
Candidate Pair Selection — Cross-Modal, Project-Scoped
For each extracted claim, Pinecone queries the project's namespace (namespace=project_id) to find the top-5 semantically similar claims across ALL modalities. A voice memo claim is compared against PDF, image OCR, and handwritten note claims equally. Critically, the search is bounded by the Pinecone namespace — no cross-project comparison ever occurs because each project is a separate namespace.
pinecone.query(namespace=project_id, vector=claim_vec, top_k=5) Pinecone serverless ANN — managed index, auto-scaled Cross-modal: voice vs PDF vs image vs handwritten
3
NLI Classification — Evidence-Grounded, Cannot Hallucinate
For each candidate pair, the model must classify ENTAILMENT, NEUTRAL, or CONTRADICTION — and it cannot output CONTRADICTION without quoting the exact contradicting text from both claims. The prompt includes modality context: "Claim A comes from a voice recording by Speaker 2 on March 15. Claim B comes from the signed PDF contract dated April 30." The model can reason about temporal and authority relationships.
Required output: {relation, evidence_a, evidence_b, conflict_type, modality_a, modality_b} Conflict types: factual / temporal / definitional / logical / cross-modal No evidence quotes = rejection, not output
4
Verification Pass — Fresh Context Window
A second independent LLM call receives ONLY the two evidence quotes — no other document content. This eliminates hallucination from context bleed. The verifier cannot invent content from the rest of the document. If first and second pass disagree, the conflict is routed to the project's human review queue (visible to Admins) rather than surfaced to Members.
Strict context: ONLY the two evidence quotes, nothing else Disagreement → project's human_review_queue in Firestore Disagreement rate is a monitored project-level metric
5
Severity Scoring & Workflow Routing
Scoring accounts for conflict type, document authority, temporal ordering, and modality. Stored in Firestore under projects/{project_id}/conflicts/{conflict_id} with full evidence trail and adjudication workflow. Members see confirmed conflicts; Admins see everything including pending review items.
CRITICAL: cross-modal conflict involving a signed document HIGH: dated factual claims, financial figures, legal obligations LOW: stylistic differences, non-binding notes Status workflow: pending_review → confirmed → dismissed → escalated

Security Hardening

Enterprise Grade
🔑 Authentication & JWT
  • Firebase Auth + short-lived JWTs: 15-minute access tokens with refresh tokens stored in httpOnly cookies. Never in localStorage.
  • Token contains user_id only: project_id is always an explicit request parameter, never baked into the token — this prevents token reuse across projects.
  • MFA enforced for admins: Project Admin role requires MFA via Firebase Auth. Optional for Members, not available for Viewers.
  • Token revocation: Refresh tokens stored in Firestore. On logout or role change, token blacklisted immediately — no grace period.
📁 File Upload Security
  • MIME validation: python-magic reads file bytes — not filename extension. Mismatches rejected at API layer before storage.
  • Size limits: 50MB per file, 500MB per project per day (configurable by Admin per project). ZIP bomb protection: reject if decompressed size >10x compressed.
  • Malware scanning: ClamAV via clamd socket in worker before any AI processing. Failed files quarantined in project-scoped GCS path, never deleted.
  • PDF sanitization: pikepdf strips JavaScript, embedded files, and metadata before any storage or processing.
  • Approval gate: If the project requires upload approval, files are quarantined in pending state even after passing all security checks — AI processing only begins after Admin approval.
💉 Prompt Injection Protection
  • Content delimiting: <untrusted_document_content> XML tags wrap all user-uploaded content in every agent prompt.
  • Project boundary in system prompt: "You serve only project {project_id}. You have access only to documents approved in project {project_id}."
  • Output validation: Check LLM output for system prompt fragments via sliding-window match. Incidents logged to audit_log and rate-limited aggressively.
  • QuerySanitizer: User queries classified for injection attempts before reaching agents. Suspicious patterns logged and rate-limited at the project level.
🚦 Rate Limiting & Abuse
  • Per-user limits: 20 queries/min, 5 uploads/min (via slowapi + Redis).
  • Per-project limits: 1,000 API calls/hr per project. Prevents a large project from starving smaller ones on the same platform.
  • Cloud Armor WAF: DDoS protection, OWASP CRS, geographic restrictions at the Load Balancer before any application code runs.
  • Cost circuit breaker: Per-project Gemini spend tracked in Redis. At 90% of daily budget, non-urgent requests queued. Admin notified before overage.
🔒 Encryption & Privacy
  • TLS 1.3 minimum everywhere. HSTS headers. No HTTP path in the infrastructure.
  • CMEK: Cloud SQL and Cloud Storage use Customer-Managed Encryption Keys. Projects can rotate their key independently.
  • Data residency: europe-west1 for GDPR-bound projects, us-central1 for US projects. Never mix regions for a single project's data.
  • Right to erasure: DELETE /projects/{project_id} purges all SQL rows, Firestore nodes, GCS files, Pinecone namespace vectors, Redis cache, and audit logs for that project, then issues a deletion certificate. Legal hold prevents this.
📋 Compliance & Audit
  • Append-only audit_log: Every document access, query, role change, approval action, and admin operation logged with user_id, project_id, role_at_time, action, resource_id, timestamp, IP.
  • Legal hold: Admins can freeze a project's data from any deletion for litigation discovery. legal_hold: true prevents all automated retention purges.
  • Attorney-client privilege: privilege_status: attorney_client on documents restricts access to specific roles and excludes from AI proactive analysis.
  • GDPR compliance: Per-project data retention policy (default 90 days inactivity → archive). Project-level deletion certificate on purge. DPA template in onboarding.

Frontend — Project-Aware Research UX

Research Grade
Design Principle: The project switcher is the most critical UX element. Users must always know which project they are in. The active project context is displayed in the header at all times. All operations — uploads, queries, conflict views, knowledge graph — are scoped to the active project. There is no accidental cross-project operation.
🔀 Project Context System
  • Project switcher in header: Always visible. Shows active project name, user's role in it, and member count. Click opens a dropdown of all your projects sorted by last activity.
  • Cmd+K command palette: Type "switch to [name]" to switch projects instantly. Shows all accessible projects with role indicator (Admin/Member/Viewer badge).
  • Context reset on switch: Switching project resets the entire workspace — search, document list, chat history, knowledge graph — to the new project's data. A confirmation prompt if there's an unsaved query in progress.
  • Role-aware UI: Buttons and actions adapt to the user's role. An upload button visible to Members is hidden for Viewers. Admin-only pages (approval queue, member management) are not rendered — not just grayed out — for non-admins.
✅ Admin Dashboard
  • Approval queue: Pending uploads shown with filename, uploader, modality, file size, and time since upload. Batch approve by type. Approve or reject individual files with optional reason (reason emailed to uploader).
  • Member management: Add by email. Assign/change role. Remove member. Transfer admin role. See last active time per member.
  • Human review queue: Low-confidence conflicts shown side-by-side with both evidence spans. Confirm (surface to project members) / Dismiss (suppress with reason) / Escalate actions. Keyboard-navigable.
  • Audit log: Filterable by user, action type, date range. CSV export. Shows: who accessed what document, when, with what query, from which IP, with what role at the time.
  • Usage & cost: Per-project Gemini API usage, document counts, storage used, active member count. Monthly cost breakdown.
📄 Document & Query Interface
  • Split-pane layout: Source document (PDF.js) left, extracted insights right. Click a citation → scrolls PDF to exact source location.
  • Conflict highlighting: Contradicting spans highlighted in red across both documents when a conflict is selected in the sidebar.
  • Grounding confidence: Shows both grounding_confidence (% claims verified against project documents) and model_confidence separately. Never one ambiguous number.
  • Upload status: Firestore real-time listener shows processing stages live — Validating → Scanning → Pending Approval → Extracting → Complete. If approval required, shows "Awaiting admin approval" with the pending indicator.
  • Voice recording: Real-time transcript as audio is captured. Shows words appearing as they are spoken. Clearly labeled with speaker diarization output once processed.
⌨ Keyboard-First Power User UX
  • Command palette (Cmd+K): Search documents, switch projects, jump to conflicts view, navigate to member management. Lawyers live here.
  • Keyboard shortcuts: Cmd+U (upload), Cmd+/ (query), Cmd+1/2/3 (density modes), Cmd+[ / ] (navigate documents), Esc (dismiss panel), Cmd+Enter (submit query).
  • Information density modes: Compact (table, 4 lines/doc), Normal (card with summary), Reading (full-width single doc). Persisted to user profile, not project — consistent across all projects.
  • Knowledge graph: Sigma.js WebGL — 10,000+ nodes without frame drops. Filter by entity type, relationship score threshold, and document modality. Temporal edge display toggle.
⚡ Performance Engineering
  • react-virtual: Virtualized document list — renders only visible rows even with 10,000 documents in a project corpus. No pagination needed for typical project sizes.
  • PDF.js with lazy page loading: Only renders pages in the viewport. A 500-page contract loads page 1 in under 1 second.
  • TanStack Query: Server state cached with stale-while-revalidate. Project switch triggers immediate cache invalidation for the previous project — no cross-project data visible during transition.
  • Chunked file upload: 5MB chunks with progress tracking and automatic resume-on-failure. Optimistic UI shows document as "Processing" in the project list before the server confirms receipt.
  • Code splitting: Project switcher, Document Viewer, Knowledge Graph, Admin Dashboard, and Approval Queue are separate JS chunks. Initial load <150KB gzipped.

Goal State at Week 16: OmniMind v2 is a production-grade, multi-project AI cognitive research platform. Users can belong to unlimited projects with independent roles. Project Admins fully control membership, uploads, and AI analysis within their workspace. Data isolation is enforced at four independent layers — the database, the application, the storage infrastructure, and the AI prompt boundary. Contradictions are evidence-grounded with mandatory human review for uncertain cases. Every factual answer is cited against project documents or marked UNVERIFIED. Deployments are zero-downtime with canary rollouts and automated rollback. The cross-modal conflict engine (voice vs PDF vs image vs handwritten) is the unique capability no competitor offers — and it works entirely within the project boundary, with no data crossing project lines under any circumstances.
